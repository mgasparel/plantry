## UI work

Plantry's UI is server-rendered Razor Pages with htmx and Alpine.js ‚Äî no Node, no bundler, no SPA. Styles live in `src/Plantry.Web/wwwroot/css/plenish.css`.

### Component library

`src/Plantry.Web/Pages/Dev/Index.cshtml` is the source of truth for **shared, reusable** UI ‚Äî the cross-cutting building blocks feature pages compose with: reusable Razor tag helpers/partials (e.g. `<field>`) and canonical CSS patterns reused across pages (e.g. cards, segmented controls, pills, steppers). It is **not** an inventory of every element in the app ‚Äî page-specific layout and one-off feature compositions live on their own feature page. It is development-only and 404s outside the `Development` environment.

Before building any UI element:

1. **Reuse first ‚Äî check the library.** If a suitable primitive already exists, compose from its canonical markup; don't reinvent a near-duplicate. Reach for existing elements when designing new pages.
2. **Extract before you repeat.** If you'd be writing the same markup a second time, or building a clearly reusable widget, extract it into a tag helper / partial / CSS pattern, add it to the library first, then consume it from feature pages. Propose genuinely new reusable components to the user before building them.
3. **Don't catalogue page-specific markup.** A new library entry is warranted only when the thing is reused across pages or is a generic primitive. Single-use layout that merely composes existing primitives stays on its feature page.

### Architecture
Plantry is a multi-tenant household inventory and meal planning platform: it ingests user-defined recipes, grocery catalogs, and consumption logs, processes them through isolated domain sub-systems (Inventory, Recipes, Catalog, Intake) using Domain-Driven Design (DDD) principles, and outputs real-time stock tracking, automated meal planning, and consumption analytics via a web UI orchestrated by a .NET Aspire host. The system enforces strict data isolation between households using a database-level Row-Level Security (RLS) connection interceptor, ensuring that all domain entities‚Äîfrom product stocks to recipes‚Äîare securely scoped to a specific household identifier. The codebase is structured around two primary runtime entry points and two documentation entry points:

### Repowise MCP Tools

This repo has the Repowise MCP server configured. The tools below answer questions `grep`/`Read` cannot. Every response carries an `_meta` envelope with `index_age_days`, `indexed_commit`, and a `stale_warning` only when the index has actually diverged from HEAD ‚Äî silence means the index is current.

**When to call which tool:**

| Tool | What only this tool answers |
|------|------------------------------|
| `get_answer(question)` | Synthesised answer with verified citations and a calibrated `retrieval_quality`. First call for "how does X work" / "why is Y like this". On low confidence returns `best_guesses` with one-line justifications instead of an empty answer. |
| `get_context(targets=[...])` | Triage card for files/modules/symbols ‚Äî title, summary, signatures, `hotspot` bit, `decision_records` titles, and `symbol_id`s to pipe into `get_symbol`. Use `include=["callers","ownership",...]` to widen. NOT for source bytes. |
| `get_symbol("path/to/file.py::Name")` | Raw source bytes for one indexed symbol with exact line bounds. Cheaper and safer than `Read` + offset math. Use the `symbol_id` returned by `get_context`. |
| `search_codebase(query, kind?)` | Find pages by concept when you don't know the file. Each result carries `search_method` (`embedding` vs `bm25` fallback). For exact identifiers use Grep ‚Äî the tool will hint when it sees one. |
| `get_why(query, targets?)` | Architectural decision archaeology ‚Äî *why* the code is shaped this way. Call before refactors or pattern divergences. Falls back to git archaeology when no ADRs exist for a file. |
| `get_risk(targets, changed_files?)` | What history says about touching these files: churn, owners, blast radius. Pass `changed_files` for PR mode ‚Üí returns a `directive` (`will_break`, `missing_cochanges`, `missing_tests`). |
| `get_dead_code(...)` | Tiered unreachable / unused-export / zombie-package findings. Run before a cleanup sprint, not before a targeted fix. |
| `get_overview(repo?)` | Architecture map for an unfamiliar repo. One-time orientation; skip on subsequent calls in the same session. |

**Composition tips:**
- `get_answer` ‚Üí if `confidence` is `medium`/`low`, follow the `best_guesses[0].file` or `fallback_targets[0]` into `get_context`, then `get_symbol` for bytes.
- `get_context` returns `decision_records` titles ‚Üí call `get_why(targets=[...])` for the rationale.
- `get_context` returns `hotspot: true` ‚Üí call `get_risk` before editing.
- PR review ‚Üí `get_risk(targets=[...], changed_files=[...])`; read the `directive` block first.

**Verify when:** `_meta.stale_warning` is present, or `retrieval_quality` is `partial`/`weak`, or `search_method` is `bm25`. Otherwise trust the response and act on it.
