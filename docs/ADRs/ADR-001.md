# ADR-001 — DDD as architectural foundation

**Status:** Accepted · 2026-06-06

## Context

Plantry has real domain complexity: FEFO consumption, freeze/thaw expiry recalculation, unit conversion, fulfillment scoring, deal matching. This is behavior, not just data.

## Decision

Model the system with DDD — aggregates, value objects, application services, and explicit bounded contexts. The domain layer holds all business rules and has no dependency on transport or persistence.

## Consequences

Every later decision is judged on whether it keeps the domain as the single source of truth. Approaches that tempt logic to drift into a second tier (a JS server, a client model, the database) are disfavored.
