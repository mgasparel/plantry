# ADR-007 — AI orchestration runs server-side in .NET

**Status:** Accepted · 2026-06-06 · Amended 2026-06-06

## Context

Several features depend on AI: receipt parsing and catalog matching (intake), and the meal planner (composes a week from pantry, expiry, fulfillment, and deals). A working POC (`/ReceiptPoc`) already does this in C# — a two-stage pipeline (raw text → structured line items → confidence-scored catalog matches) calling a model over an OpenAI-compatible `ChatClient`. The question is only *where* orchestration lives, not whether to build it.

## Decision

AI orchestration is a **server-side concern inside the .NET application layer**. Prompt construction, model calls, JSON parsing, and confidence handling live behind application services — the same services the UI and any future MCP/REST transport call. The AI is treated as an untrusted external function: its output is a *proposal* that flows into a review-then-commit step (the import review form, the meal-plan review), never a direct write to the domain. The model talks to no database and invokes no domain logic directly; it returns structured data that the application layer validates and maps onto aggregates.

## Consequences

- Consistent with ADR-001/004: no second tier. The API key is held server-side (already reflected in Settings §7f), and the client never calls a model directly.
- Semi-structured model output lands in `jsonb` (ADR-003) on a staging/proposal record until the user confirms; confirmation is what writes journal entries / catalog rows.
- The meal planner, if it needs tool-style access to inventory, uses the in-process application services (or the Plantry-owned MCP surface per ADR-006), not a separate agent runtime.

## Open / deferred

- **Image → text step** — Claude vision (image straight to structured items) vs. a dedicated OCR pass feeding the existing text pipeline. The POC starts from receipt *text*, so either upstream choice plugs in without reworking stages 1–2.
- **Model / provider selection** — the POC's OpenAI-compatible `ChatClient` can point at multiple providers. Final model(s) per task (parse, match, plan) undecided.

## Amendment · 2026-06-06 — key storage and the proposal record (DATA-MODEL DM-7, DM-15)

Concretising "the API key is held server-side":

- The key is **per-household and user-entered**, stored as `household_settings.ai_api_key_encrypted` (`bytea`), **encrypted at rest** via ASP.NET Core Data Protection. It is decrypted server-side only and **never serialized to the client** (the settings form shows a masked "key set ••••"). This reconciles user-supplied keys with "held server-side, never exposed to client" — the client supplies the key once over TLS and can never read it back.
- The "staging/proposal record" where semi-structured model output lands is concretely the Intake context's `import_session` root with `import_line` children: the raw AI payload is quarantined per row in `raw_parse` (`jsonb`) plus a typed `suggested_confidence`, and **only user-resolved typed fields commit** to Catalog/Inventory/Pricing. The same review-then-commit shape applies to the meal-plan proposal (Phase 2).
