# ADR-011 — Single consumption primitive (Inventory `Consume`)

**Status:** Accepted · 2026-06-06 · Amended 2026-06-06

## Context

Stock leaves the pantry through several triggers: manual "I used some" (SPEC §1c), marking a specific lot gone/spoiled (§1b), cooking a recipe with per-ingredient swap/skip (§4c), and discarding expired items (§1d). Modeling each as its own write path would scatter FEFO ordering, unit conversion, and journaling, and would make consumption history (§1b) and waste analysis (VISION Phase 4) inconsistent.

## Decision

Exactly **one** consumption primitive on the Inventory aggregate root:

`ProductStock.Consume(quantity, unit, reason, sourceRef?)` — converts the requested unit (Catalog conversions), orders that product's lots FEFO, deducts across lots, writes immutable `StockJournalEntry` rows, and reports any shortfall. It can optionally target a specific lot instead of FEFO (for "this carton is empty/spoiled"). It is the *only* operation that removes stock; Inventory knows nothing about recipes or substitutions.

- **`reason` is a typed taxonomy** that critically distinguishes **Consumed (used)** from **Discarded (wasted/expired)** — plus Correction. Waste is impossible to reconstruct later if every removal is logged generically, and minimizing waste is a stated product goal (VISION).
- **Cook is a Recipes-context orchestration**, not an Inventory operation: it scales ingredient quantities by servings, applies swap/skip, then issues one `Consume(reason: Cook, sourceRef: cookEventId)` per resolved ingredient and writes a `CookEvent` grouping them. A **swap** changes *which* `ProductStock` receives the command (and the quantity, if units differ); a **skip** omits the command. Substitution logic lives entirely in cook — Inventory never hears the term "substitution."
- **Manual consume, mark-lot-gone, and throw-out-expired** are direct `Consume` calls with reason Manual or Discarded.

## Consequences

- The stock journal is the single source of truth for every removal and its reason — consumption history and waste/trend analysis read the same log.
- Cook touching N ingredients fans out to N `ProductStock` aggregates; per-ingredient quantities are fixed at the review step (§4c), so a shortfall is an explicit user choice, not a commit-time surprise. Whether the fan-out commits as one transaction or N is an implementation detail (the bounded multi-aggregate exception noted in ADR-010).
- New removal triggers (bulk pantry cleanout, etc.) add a `reason`, not a new write path. Inventory's vocabulary stays "consume this product, this much, for this reason."

## Amendment · 2026-06-06 — journal scope and taxonomy (DATA-MODEL DM-13, DM-14)

Schema modeling sharpened what the journal is, and isn't, the source of truth for:

- **"Single source of truth for every removal" → every *quantity movement*.** The append-only `StockJournalEntry` records signed quantity deltas: removals (this ADR's `Consume`) **and** intake. Quantity-*neutral* lot-state transitions — transfer, freeze/thaw, open — change location/state and the materialized `expiry_date` but write **no** journal row; they are **current state on the `StockEntry`**. Full move-history is deferred to a `stock_entry_event` audit log, added only if v1 needs it. So the journal is authoritative for *how much moved and why*, not for *every lot-lifecycle event*.
- **`reason` and `source` are separate axes.** `Consume`'s removal taxonomy is `Consumed` / `Discarded` / `Correction`. The journal's `reason` column adds **`Purchase`** for intake (written by the Intake→Inventory record-purchase path, not by `Consume`). Orthogonally, a `source_type` (`Intake` / `Manual` / `Cook` …) with a `source_ref` records *what triggered* the entry. The ADR's earlier shorthand ("reason Manual", "reason: Cook") is reclassified: Manual/Cook/Intake are `source_type` values; the `reason` stays the why-taxonomy above.
- **Lots are retained, never hard-deleted.** A depleted lot (`quantity = 0`) keeps its row with `depleted_at` set, because the append-only journal's FK to `StockEntry` requires every historical lot to remain live.
