# ADR-013 — Intake review form: targeted out-of-band swaps over full-region re-render

**Status:** Accepted · 2026-06-12 · Validated by spike plantry-5l8 (2026-06-12) · Amended 2026-06-18, 2026-06-22 · **Superseded-in-practice on island surfaces by [ADR-020](ADR-020.md)** (pattern retained for any htmx derived-view surface)

Resolves the ADR-005 watch item ("the import review form is the worst case… to be pressure-tested against Alpine"). The form stays within htmx + Alpine; the escape hatch is **not** triggered.

## Context

The intake review form (`src/Plantry.Web/Pages/Intake/`) renders one `ImportSession` as a list of resolvable line rows plus several **page-level aggregates** derived from those lines: the filter-chip counts (All / Needs review / Ready), the progress bar, the commit bar (confirmed count + value), and the receipt total. The aggregates must stay in lock-step with the rows as the user confirms, dismisses, and restores lines — that synchronisation *is* the product feel.

The current implementation keeps them in sync with one mechanism: **every row action re-renders the entire `#rev-body` region** (`Review.cshtml.cs::RowResultAsync`). This welds two distinct operations together — *"update the aggregates"* and *"repaint every row"* — so they can only happen as a unit. The result is a false dilemma with no good resolution inside the mechanism:

- Keeping aggregates current requires repainting all rows, which **destroys the client (Alpine) state of every other row** — any open edit drawer with unsaved quantity/expiry/product edits is wiped on each confirm.
- Preserving a row's drawer requires a row-local swap, under which the chips, counts, progress and total **drift** from the line states.

This is the source of the juddery UX and the reason incremental fixes thrash: there is no winning move while the two operations are coupled. Compounding it:

- **Derived state is computed in ~four places** — receipt total in `Review.cshtml`, progress and commit bar in the code-behind, chip counts in `_ReviewBody.cshtml`. They agree *only because* the full repaint renders them together. Break the repaint and there is no single thing that keeps them consistent.
- **Two competing row models live in the tree.** `_ReviewRow.cshtml` (server-computed prefill + full-region repaint) is canonical; `_ImportLineRow.cshtml` (client `draft` state + per-row `outerHTML` swap + `hx-vals="JSON.stringify(draft)"`) is an earlier, opposite strategy now used only by the Dev component library. Two contradictory answers to "where does row state live" make the architecture impossible to reason about locally.

htmx already ships the right primitive, and the form already uses it **once, as an afterthought**: the commit bar returns via `hx-swap-oob` (`_ReviewBody.cshtml`). Out-of-band swaps let a single response carry the changed row *plus* independent fragments for each aggregate, each landing in its own place — without touching any other row.

## Decision

**Server remains the single source of truth** for line state and every aggregate (reinforces ADR-004 / ADR-005). The client holds only ephemeral UI state. We separate "update the aggregates" from "repaint the rows" by making targeted OOB swaps the **primary** delivery mechanism, not the exception.

### 1. The row-action response contract

Every row-mutation handler (`SaveLine`, `DismissLine`, `RestoreLine`) returns a fixed fragment **bundle**, never a full-region repaint:

- the **one changed row**, swapped in place (`outerHTML` on `#import-line-{id}`); and
- an **OOB fragment per aggregate projection**: filter chips + counts, progress bar, commit bar, receipt total.

This contract is the invariant the form was missing. It is written down so that any future change — by a human or an agent — has a concrete target ("return the row + the projection OOB set") and cannot silently break global consistency by editing one handler.

### 2. One projection builder

All page-level aggregates are derived in a **single** place from the session and rendered as independently-addressable OOB fragments. The four scattered computations collapse into one "build the page projections" call reused by both the initial render and every row response. "Keep the aggregates in sync" becomes a single function, not an emergent property of repainting.

### 3. Rows are stable DOM nodes; section membership is a reactive projection

Because rows are no longer repainted as a group, a confirmed line must move from "Needs review" to "Matched & ready" **without** a server repaint. A row is therefore a **stable DOM node that carries its own status**; which section it appears under, and which filter chip shows it, are reactive (Alpine `x-show` on status + active filter), not a function of server-rendered DOM structure. Prefer a **flat row list** with client-driven section headers and grouping over server-rendered section containers, so a status change never needs to relocate a node across containers. The server-side partitioning is retained only to render the **initial** page and to supply authoritative counts in the OOB fragments.

### 4. Optimistic aggregates are an optional, strictly-bounded thin slice

For zero-latency feel — chiefly a running total that updates *while the user types in a drawer*, before save — a **single** Alpine store may cache line statuses/values and recompute aggregates client-side on interaction. This store is a **latency-hiding cache, never a parallel domain model**: every server response's OOB fragment is authoritative and overwrites it, so the server wins on every round trip and the two cannot durably drift. This slice is additive; the form is fully functional without it. If this store ever grows toward owning domain logic (validation, conversions, persistence-shaped decisions), that **is** the ADR-005 escape-hatch signal and must be recorded as an amendment, not absorbed silently.

### 5. One canonical row model

`_ReviewRow.cshtml` is the single row implementation. `_ImportLineRow.cshtml`'s competing client-`draft`/per-row-swap strategy is retired from the feature path (kept, if at all, only as a Dev component-library example, clearly labelled non-canonical). The tree carries exactly one answer to "where does row state live."

## Consequences

- **Fluid UX.** Addressing a row updates only that row plus the aggregates; every other row's open drawer and unsaved edits survive. The judder caused by repainting the whole list is gone.
- **A real invariant for local change.** The response contract + single projection builder give humans and agents a target to satisfy, which is the concrete fix for "Claude keeps proposing changes that break something else": local edits now have a defined global obligation instead of relying on a brute-force repaint to hide drift.
- **More wiring per handler, centralised.** Each handler assembles a fragment bundle rather than returning one body. The cost is contained in the projection builder and a shared "row response" helper, not spread across handlers.
- **A little grouping logic moves to Alpine.** Client-driven sectioning is the price of stable row nodes; mitigated by keeping the server as the source of the initial render and of all counts. This stays well within ADR-005's "Alpine for client-side local state" remit.
- **ADR-005 watch item closed without the escape hatch.** The worst-case screen is delivered in htmx + Alpine. The contingency (web component / Lit) remains untriggered; the optimistic-store boundary in §4 is the tripwire that would reopen ADR-005 if crossed.
- **The two-row-model drift is resolved**, removing the contradictory precedent that made the form hard to reason about.

## Implementation notes (non-binding)

- Split the current `_ReviewBody` into addressable fragments: the header/aggregates (chips + progress) and the row list, so the header can be delivered OOB independently of the rows.
- Drop the server values currently interpolated into Alpine JS literals (`x-data='{ v: @qtyInitJs }'`, the expiry `adj()` one-liner) into the JSON-serialised `x-data` the row state already uses — a correctness cleanup that this restructure is the natural moment to make.
- `RowError` / `HX-Retarget` already swap a single row out-of-band on validation failure; that path is consistent with this contract and largely stays as-is.

### Bundle composition (Intake)

The row-action response is assembled by `_ReviewRowOobBundle`, composed of five partials:

| Partial | Role | Stable DOM id |
|---|---|---|
| `_ReviewRow` | One changed row — primary swap body (`HX-Retarget: #import-line-{id}`, `HX-Reswap: outerHTML`) | `#import-line-{id}` |
| `_ReviewChipsOob` | Filter chip bar + counts | `#rev-chips` |
| `_ReviewProgressOob` | Progress bar | `#rev-progress` |
| `_ReviewCommitBarOob` | Commit bar (confirmed count + value) | `#commit-bar` |
| `_ReviewReceiptTotalOob` | Receipt total | `#rcpt-total` |

Each OOB partial carries `hx-swap-oob="outerHTML:#stable-id"`. `BuildProjections()` is the single function that computes all four aggregate counts, used identically by the initial page render and every row-action response.

## Amendment · 2026-06-12 — spike plantry-5l8 outcomes

The throwaway spike (`spike/adr-013-oob-review`, findings in `ADRs/notes/adr-013-spike-findings.md`) proved the contract end-to-end on the quick-confirm path and settled the two questions §3 and §4 left open. The decisions below are now binding on the production conversion.

- **§3 sectioning — Option A confirmed (flat list + Alpine `x-show`), with one hard requirement.** Server-rendered section containers + OOB relocation (Option B) is rejected: `hx-swap-oob` replaces in place and cannot move a node between containers without a delete-then-insert race, and a row crossing containers destroys its Alpine component — the exact thing we are eliminating. **Binding requirement:** the `data-status` attribute that drives section membership **must live on, or inside, the element swapped by the OOB row swap (`#import-line-{id}`)**. If it sits on an outer wrapper, a confirmed row's section membership is stale in the DOM until reload. The spike's wrapper sat outside the row id; the production conversion must move it inside.
- **§4 optimistic store — not built.** The server round-trip on save is fluid enough; the aggregates only have meaning after a save commits. No client-side store. The §4 boundary stands as a dormant tripwire: it is reconsidered only if the product later wants a live-while-typing total, and any such store must be overwritten by the server's OOB fragment on every round trip.
- **Section-header counts come from the server (the OOB chips fragment), not client DOM-counting.** The spike used `$root.querySelectorAll()` to count visible rows for the section headers; that is rejected as a smell. Counts are a server projection delivered OOB, consistent with the single-projection-builder decision.
- **The contract is enforced by a test.** `ReviewBoundaryTests` was flipped from asserting the old full-repaint output to asserting the contract (`HX-Retarget` header + OOB fragment ids in the body). This assertion is carried forward and extended to every converted handler — it is the guardrail that keeps future iteration from silently reverting to a full repaint or dropping a projection.

## Amendment · 2026-06-18 — the contract is a cross-feature primitive, not an Intake-local pattern (plantry-c2c)

**This ADR described a pattern but did not make it transferable, and that gap had a cost.** When the meal planner grew its own page-level derived view — the P3-5 insights rail (`plantry-6si`) — the same "derived projection drifts after a localized mutation" bug reappeared: `OnPostAssign`/`OnPostClear` returned a bare cell and left the rail stale, while six sibling handlers happened to refresh it via a full-grid swap. The fix was a re-derivation of *this ADR's* contract (a single mutation funnel that emits the changed element + the rail projection). Notably, **nobody reached for ADR-013**: the implementing worker, the reviewer, and a three-pass Opus critic all flagged the symptom and none cited the existing decision. A pattern that lives only as prose plus a feature-local test is invisible exactly when a different feature needs it.

Corrective decision (binding):

- **The response contract is enforced through a shared primitive, `OobContract` (`tests/Plantry.Tests.Web/Infrastructure/OobContract.cs`).** `AssertCarriesProjections(body, …ids)` is the mechanism-agnostic core: a mutation response must carry a fresh element for every named derived-view projection, whether delivered inline (full-region swap) or out-of-band (targeted swap). `AssertRetargets` / `AssertNoFullRepaint` capture the Intake row-swap specifics.
- **Both features now assert through this one primitive.** `ReviewBoundaryTests` (Intake: `rev-chips` / `rev-progress` / `commit-bar` / `rcpt-total`) and `InsightsRailFragmentTests` (MealPlanning: `plan-rail`, on both the cell-swap and full-grid paths) call `OobContract`, so "a mutation carries its projection bundle" is one enforced rule across contexts.
- **General rule for any feature with page-level derived views:** route every mutation handler through a single funnel that emits the projection bundle, and pin it with `OobContract`. This is now the default any new such feature mirrors — the structure, not this prose, is what transfers.
- **Meta-lesson (recorded deliberately):** an architectural pattern needs an *enforcing artifact* (a shared primitive + a test), not just a decision record. An ADR without one is decoration that the next agent — human or AI — will not find under load, because work is navigated by what fails red, not by what is written down.

## Amendment · 2026-06-22 — superseded-in-practice on island surfaces by [ADR-020](ADR-020.md) (bead plantry-2zvm)

[ADR-020](ADR-020.md) moves the two surfaces this contract governs — the Intake review form and the MealPlanning insights rail — onto a client-side **reactive runtime**. On those surfaces the OOB-bundle mechanism is **retired**: the five `*Oob` partials, `BuildProjections()`, and the `OobContract` assertions are replaced by reactive `computed` values that derive each aggregate from state the island already holds. The drift this ADR fights becomes structurally impossible rather than test-enforced.

This is **supersession-in-practice, not retirement**, for three reasons:

- **The principle survives.** "A mutation must refresh every derived projection, and the server stays authoritative for the underlying state" still holds — ADR-020's runtime enforces it *by construction* instead of via a funnel + test. ADR-020 is the new enforcing artifact for exactly the property this ADR's 2026-06-18 meta-lesson demanded.
- **The pattern still governs htmx surfaces.** Any *new* server-driven page that grows a page-level derived view — and is not an enumerated ADR-020 island — must still use this contract (single mutation funnel + `OobContract`). An agent building a hypermedia derived-view surface should land here, not on ADR-020.
- **This ADR's history is the argument for ADR-020.** The 2026-06-18 meta-lesson — *work is navigated by what fails red, not by what is written down* — is precisely why a reactive runtime (which makes drift impossible) is preferred over a hand-maintained contract (which a worker, a reviewer, and a three-pass critic already failed to find once).

If the islands later subsume **all** derived-view surfaces, this ADR is re-evaluated for a full `Superseded` status. Until then it stays `Accepted` and authoritative for the htmx path.
