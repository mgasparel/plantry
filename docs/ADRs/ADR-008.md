# ADR-008 — Authentication and household multi-tenancy built in from day one

**Status:** Accepted · 2026-06-06 · Amended 2026-06-06

## Context

Earlier framing treated the app as single-user with auth merely gating access. The product is explicitly household-centric (see VISION), and multiple users sharing one home is the real usage pattern. Retrofitting a tenancy boundary onto a single-user schema later is expensive and error-prone; since we are auth-gating anyway, the boundary is cheap to establish now and costly to add later.

## Decision

The **household is the tenant**. A household owns the pantry, catalog, recipes, shopping list, locations, deals, and meal plan; **users are members of a household** and share all of it. Every domain row is scoped by `household_id`, enforced at the repository/query layer so a request can only ever see its own household's data. Authentication identifies the *user*; authorization scopes them to their *household*. Plantry ships with real auth (login, sessions, password handling) from the first release rather than a stubbed single user.

## Consequences

- `household_id` is a mandatory part of aggregate identity and of every repository read/write.
- Single-user is simply a household of one; no separate code path. Inviting a second member is additive, not a migration.
- **Deliberately deferred:** per-member **roles/permissions** (owner vs. member) and per-user preferences. v1 is flat — every member has equal rights within the household.
- Auth mechanism specifics (cookie sessions vs. tokens, password reset, invite flow) are an implementation detail to settle during Phase 1, not a foundational ADR.

## Amendment · 2026-06-06 — tenancy mechanism

The tenant boundary is realized as **shared database, shared schema, `household_id` discriminator** — *not* database-per-tenant. Plantry's tenants are many and small (a household is a few people with a modest pantry), which is exactly the regime where db-per-tenant's costs dominate with no offsetting benefit: migration fan-out across N databases, connection-pool pressure from many small tenants, per-signup provisioning (create db + migrate + seed), and N backup streams against ADR-009's deciding "one backup story" rationale.

**Postgres Row-Level Security**, keyed on a per-request session variable (`SET app.household_id`), backs up the repository-layer scoping as defense-in-depth — a query that forgets its `WHERE household_id` still physically cannot cross tenants.

Physical isolation, if a future enterprise/regulated customer ever requires it, is delivered by a **separate deployment** (cf. ADR-012's self-contained stack), not by sharding databases inside one cluster; the household-scoped repositories make that a deployment choice, not a model change.

## Amendment · 2026-06-06 — auth via ASP.NET Core Identity (DATA-MODEL DM-6)

The "auth mechanism specifics … implementation detail to settle during Phase 1" is now settled: **authentication, membership, and sessions are delegated to ASP.NET Core Identity.** We do *not* hand-roll user, password, or session tables — Identity owns email, password hashing, security stamp, lockout, and the auth cookie/session machinery, with its `AspNet*` tables living in the `identity` schema.

- `IdentityUser<Guid>` is extended with `HouseholdId` (→ `household`) and `DisplayName`.
- **Membership = `HouseholdId` on the user.** v1 is flat (no roles, consistent with the deferral above): a user belongs to exactly one household, so there is **no join table**. A `user_household` many-to-many is the deferred shape if a user ever joins multiple households.
- Identity has no invite concept, so we own the two-step invite (`household_invite`: household issues invite → user accepts) and the `household` / `household_settings` tables.
