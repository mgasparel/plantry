# ADR-019 — Skills until they break: keep agentic tooling minimal, escalate on concrete triggers

**Status:** Accepted · 2026-06-19

## Context

ADR-018 commits to fully agentic engineering. The implementation runs on Claude Code skills/agents (`implement-ticket-worker`, the driver, the critic, `groom`/`triage`, the planned `ci-fix-worker`) plus **beads** as the work ledger and local tooling. As agent frameworks mature, the question recurs: adopt a framework or stay on bespoke skills? Archon (coleam00/archon) prompted this evaluation — and it is broader than a workflow engine: a knowledge-base + project/task-management + MCP backend.

The pull toward a framework is real: we hand-roll orchestration (the serial driver, the worker, the critic loop, the CI auto-fix reconcile). But the setup also has hard shape that constrains the choice:

- **Local-only AI** — a Claude Code subscription, not an API plan (ADR-016/018): any framework that drives the model itself via API is out; Claude Code is the executor.
- **Portability / no lock-in** — a recurring value (ADR-016): the current stack is markdown skills + git-synced beads + plain CI, with no stateful service on the critical path.
- **Task management is already owned** by beads (git-native via Dolt over `refs/dolt/data`, dependency graph, atomic claims), and **codebase knowledge** by Repowise (MCP). A framework's headline features largely overlap these.

## Decision

**Use the simplest tooling that reliably does the job — Claude Code skills + beads — and keep using it until it breaks.** Escalate to heavier machinery only when a concrete trigger forces it, never speculatively because a more full-featured option exists.

This is a **policy, not a verdict on frameworks.** "Break" means concrete, observable triggers — for example:

- **Orchestration sprawl** — the driver/skills become genuinely hard to reason about or maintain.
- **A capability skills cannot reliably provide** — a real, recurring need that markdown skills + MCP cannot meet.
- **Reliability failures in the loop** — the bespoke machinery proves flaky in practice.

Discovering that a framework is *more full-featured* is **not** a trigger. When a trigger does fire, prefer adding the **single missing capability** (e.g. a targeted MCP server) over adopting a whole framework, to preserve portability.

## Consequences

- **The stack stays minimal and portable** — no stateful service on the critical path of how we ship; consistent with ADR-016's portability stance.
- **The escalation bar is concrete, not taste.** "Should we adopt X?" is answered by checking against the triggers above rather than re-litigated each time a new tool appears — which is the point of writing this down.
- **Better as a showcase** (ADR-018): a transparent skills + beads setup demonstrates agentic engineering from primitives rather than hiding the mechanics behind a framework.
- **We accept the cost of hand-rolling some orchestration** (the driver, `ci-fix-worker`, the reconcile loops) while it stays simple and serial — that cost is precisely the thing being watched as a trigger.
- **Archon (and similar) evaluated and deliberately deferred** — a considered pass under current constraints, not a rejection of frameworks in general.
