# ADR-018 — Fully agentic engineering

**Status:** Accepted · 2026-06-19

## Context

Plantry is built primarily by an automated, agent-driven pipeline rather than by hand. This has been true in practice from early on — the beads issue ledger, the `implement-ticket-worker`, the pre-flight critic gate, the serial-merge workflow — but it has only ever existed as *mechanics* (in `AGENTS.md` and the `.claude/skills`), never as a stated principle with a rationale. As a result the intent behind the machinery is invisible: decisions like "file the CI/CD work as beads for the pipeline to implement" (ADR-016) or "make CI an authoritative gate" only make sense in light of it, yet a reader (human or agent) cannot find the principle they serve.

Plantry is also intended as a **reference example of agentic engineering**. *How* the work is done is part of the project's purpose, not merely a means to the product — which is itself a reason to write the principle down rather than leave it implicit.

## Decision

**Fully agentic engineering is a foundational principle of this project**, peer to ADR-001 (DDD as the lens for the product). Concretely:

- **The implementation work that humans plan is carried out by the automated pipeline by default.** Humans own *intent* — product and architecture decisions, ADRs, issue specifications, and review/approval. The automation turns planned issues into implemented, gated, merged code. Beads is the seam: planned work lives as issues; the pipeline claims, implements, gates, and closes them.

- **It is a principle, not a guardrail.** Nothing prevents a human from editing code, and no such mechanism is wanted. When a task needs human judgement, or the automation is stuck, a human steps in — pragmatically, without ceremony. "A human wrote this by hand" is never a violation to be litigated; it is an allowed pragmatic choice.

- **Pragmatism over dogma, including about the process itself.** The pipeline is a means, not an end. When a process stops serving the work, we change the process rather than force the work through it. The principle is aspirational and mutable, judged by whether it produces good software well — not by adherence.

- **Quality is enforced by gates, not by trust.** Because the author is automated and the human-in-the-loop is deliberately thin, the pre-flight gate (build → full test suite incl. E2E → an independent Opus critic) and CI (ADR-016) are what make "agents do the work" safe. The gate is non-negotiable and applies regardless of who — or what — authored a change.

## Consequences

- **The existing machinery gains a stated "why".** `AGENTS.md` (operational mechanics), the skills (`implement-ticket-worker`, the critic, pre-flight, `groom`, `triage`), and the CI-as-gate decision (ADR-016) are all expressions of this principle. Filing even *infrastructure* work — the CI/CD epic itself — as beads for the pipeline to implement is the principle applied, not an exception to it.

- **The pre-flight / critic gate carries the weight a human reviewer would** in a conventional shop. It is therefore held to a high standard and is itself a first-class, improvable part of the system, documented alongside the workflow rather than treated as fixed scaffolding.

- **The gate is author-agnostic by design.** The CI merge gate and merge queue (ADR-016) exist partly so a human's occasional manual change is held to exactly the same bar as agent work — the gate does not care who authored the change. This is what lets the principle stay non-dogmatic without lowering quality.

- **Process is versioned like code.** Workflow changes — e.g. the move to PR-based CI gating, or retiring the parallel orchestrator in favour of a serial model — are expected evolution, recorded in the engineering-workflow doc and in ADRs, not silent drift.

- **The project doubles as a worked example.** Process decisions are documented so the practice is legible to outside readers, not merely functional for us — reinforcing the OSS-bound posture (ADR-016).

## Note

The concrete mechanics — the `bd ready → worker → pre-flight gate → PR → merge` loop — and the in-flight changes to it (PR-based gating, reconciling `AGENTS.md`'s direct-push rule with branch protection, retiring `pipeline-orchestrator`) live in `docs/Engineering/agentic-workflow.md` and the ADR-016 rollout, not here. This ADR records the principle; that doc records how it currently runs.
