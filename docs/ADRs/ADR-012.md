# ADR-012 — Deployment & runtime topology (Docker homelab, Aspire app model)

**Status:** Accepted · 2026-06-06

## Context

ADR-002/003/009 settled .NET + PostgreSQL + in-DB binaries; the runtime/deployment shape was still open. The target is self-hosted: **Docker in a homelab** running the app, the database, and any supporting services together. Local development uses **.NET Aspire** as the app-model/orchestration layer. A clean cloud deployment is wanted eventually but is not yet defined.

## Decision

Plantry ships as a **self-contained container stack** — the .NET web app, PostgreSQL, and any supporting services (e.g., the async email-intake processor, SPEC §2b) composed together and run in the homelab via Docker. **Aspire is the single app model:** its AppHost describes the services and their dependencies (Postgres, the web app, background workers) and drives local run/debug, and the same topology maps onto the homelab stack. This is a direct expression of the modular monolith (ADR-010): one deployable app plus its datastore and helpers, not a fleet of services.

## Consequences

- Reinforces ADR-008 (incl. its tenancy amendment): the homelab deployment is effectively a single household (of N users) with physical isolation coming from the deployment itself; the shared-schema + `household_id` + RLS model is unchanged and still supports multi-household if a hosted offering arrives.
- Aspire defines service wiring **once** for both local and homelab, so there is no separate dev/prod composition to drift.

## Open — cloud story

A hosted, multi-household deployment is deferred. Aspire publishes cleanly to a container host (its first-class path is a managed container platform; Kubernetes and Docker Compose are alternative publish targets) — the specific target is chosen when a hosted offering is actually on the roadmap, not now. The container stack keeps those options open.
