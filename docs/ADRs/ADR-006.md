# ADR-006 — No GraphQL; REST + read models (and any MCP) over Plantry's own application services

**Status:** Accepted · 2026-06-06

## Context

Under ADR-004 there is no public API today. Should one ever be needed (e.g., a future native client), the style still matters and was reasoned through now.

## Decision

If an HTTP API is introduced, use **REST designed around aggregates** (aggregate root = resource; commands = POST/PATCH on the resource), plus **dedicated read-model endpoints** for composed views (e.g., recipe fulfillment) — CQRS-lite. **Not GraphQL.** The same rule governs **MCP**: any MCP server Plantry exposes is a thin transport over Plantry's own application services — never a connection to a third-party system (in particular, **not Grocy**). Plantry is a ground-up product; it does not wrap or proxy another inventory backend.

## Consequences

- REST maps ~1:1 onto DDD: the aggregate root is the only door in, and clients cannot reach inside an aggregate because no endpoint exposes it. GraphQL hands clients a traversable graph that crosses aggregate boundaries and leaks internals.
- Composed views (pantry aggregation, fulfillment scoring) are **read models** carrying domain logic server-side, not client-assembled graph queries.
- GraphQL's benefits (many heterogeneous clients, client-driven payload shaping) are ~zero here: one client, written by the same person, in the same repo. Its costs (schema, resolvers, N+1, lost HTTP caching, per-field auth) are not worth paying.
- An MCP surface, if built (e.g., to let the AI planner operate over inventory), calls the *same* application services as the UI. There is one domain; transports differ. This keeps ADR-001 intact — no business logic leaks into an integration layer.
