# ADR-016 — CI/CD pipeline and release process (build → GHCR → Compose; CI as a serial merge gate)

**Status:** Accepted · 2026-06-19 · supersedes [ADR-012](ADR-012.md)

## Context

ADR-012 settled the runtime *shape* — a self-contained Docker stack with .NET Aspire as the app model — but left undefined how committed code becomes a running container, and how Continuous Integration relates to the project's automated development workflow.

The state at the time of this decision:

- **CI exists but does not gate.** `.github/workflows/ci.yml` builds in Release and runs unit, architecture, and integration tests with a coverage check. It triggers on `push` to `main` and `pull_request` targeting `main`.
- **The dev workflow merges directly to `main`, locally.** Work is implemented on an `issue/<id>` worktree branch, gated *locally* by a pre-flight (build → full solution test **including E2E** → Opus critic), then merged to `main` with a local fast-forward and pushed. The `issue/<id>` branch is never pushed and no PR is opened.
- **Therefore CI is a post-hoc tripwire, not a gate.** It runs *after* the merge already landed on `main`, the PR trigger never fires, and it is strictly weaker than the local pre-flight (CI skips E2E). It cannot block a bad merge.

We want CI to be a real gate ahead of deployment; a concrete path to deploy; portability (avoid lock-in, with tradeoffs flagged rather than hidden); and room for third-party self-hosting / OSS, which is **not yet decided** but should not require re-architecture.

**Decisive constraint:** development is **strictly serial** — one issue at a time, through merge, with no parallel work streams, by choice. This eliminates the parallel-merge race that would otherwise justify a merge queue and asynchronous reconciliation.

## Decision

This ADR supersedes ADR-012 and is now the authoritative record of Plantry's deployment topology. ADR-012's core shape is retained: Plantry ships as a **self-contained container stack** — the modular monolith (ADR-010) as one deployable plus its datastore and helpers — and **.NET Aspire remains the single app model for *local development***. What follows refines how that stack is built, deployed, gated, and kept safe.

**1. Build artifact — OCI images to GHCR.** `Plantry.Web` and the migrator (ADR-017) build into OCI container images via a hand-written multi-stage `Dockerfile` (transparent and builder/registry-agnostic, preferred over the .NET SDK's container target for legibility) and are pushed to GitHub Container Registry. GHCR is a standard OCI registry; it is the *only* GitHub coupling on the deployment side and is trivially swappable by changing the image prefix. Image references are injected into Compose via a variable (`${PLANTRY_WEB_IMAGE}`), never hard-coded.

**2. Deploy target — a single host running Docker Compose.** Production is one Linux host (a VPS — a *remote* homelab, consistent with ADR-012) running `docker compose`. CD performs: build/push image → `ssh` to host → `docker compose pull` → run the migrator one-shot (ADR-017) → `docker compose up -d` → health probe. Production uses a **hand-maintained `docker-compose.prod.yml`**, not the Aspire-generated composition — see *Consequences* and the ADR-012 amendment.

**3. CI as the authoritative merge gate — serial, PR-based, behind a merge queue.** The implementation worker pushes its `issue/<id>` branch and opens a PR; the merge step uses `gh pr merge --auto`; branch protection on `main` requires the CI status check **and routes merges through a GitHub merge queue**. Branch protection applies to everyone — agents *and* manual changes go through a PR; nothing is pushed directly to `main`. The agent workflow is inherently serial (one PR in flight), so the queue normally runs at depth 1 and adds no throughput cost. Its job is purely **defensive**: in the one case the workflow does not serialize — a human pushing a manual change while an agent's PR is open — the queue re-validates each PR against the actual `main` it will land on, so a green-but-stale PR cannot break trunk. We do not optimize for that interleaving, but we refuse to let it break `main`.

> **Amendment (2026-06-20):** the **merge queue is not implementable on the current repository.** GitHub merge queues are unavailable on personal-account (user-owned) repos, and `mgasparel/plantry` is user-owned. The rest of Decision 3 stands — PR-based, CI-gated, serial, branch-protected. The merge queue specifically is **deferred until the repo is org-owned** (which the OSS path in Decision 5 / Phase 5 entails); at that point, add the `merge_group:` trigger to `ci.yml` and enable "Require merge queue". Until then the defensive job the queue would do is handled by **branch protection's strict "require branches up to date" setting plus the `pipeline-orchestrator` mergeability guard**, which parks a behind/conflicting PR (`merge-conflict` / `merge-conflict:behind`) rather than letting a stale-but-green branch land. See `plantry-ctwa` (closed won't-do) and `plantry-ljmi`.

**4. The full test suite, including E2E, runs in both places.** The local pre-flight stays **full** (build → all tests incl. E2E → Opus critic) as a fast pre-filter that keeps obvious breakage out of CI; CI re-runs the same suite, **including E2E** (Docker-in-CI), as the authoritative gate. This double-runs E2E and adds latency to the serial loop — an accepted cost for a CI gate that is never weaker than the local one. **CI runs no AI step:** Plantry is on a Claude Code subscription, not an API plan, so the Opus critic runs only in the local pre-flight (via the Claude Code CLI) — CI is build/test/E2E/coverage only, and the review gate is enforced by process (the worker won't open a PR unless the critic passed), not by CI. See ADR-018 and `docs/Engineering/agentic-workflow.md`.

> **Amendment (2026-06-20):** the slow suite **no longer runs on every PR.** E2E + integration + coverage move to the **release-tag boundary**, and **production deploys only from a release tag** gated by that suite — not on every `main` merge.
>
> *Why.* At a high PR rate the per-PR E2E re-run (~7 min × every PR, doubled by the push+PR overlap) became the dominant Actions-minute cost — measured at ~82% of every CI run, enough to burn the monthly allotment in a day. The double-run was only ever justified as "a CI gate never weaker than the local one." But the local pre-flight (Decision 4, above) *already* runs the full suite incl. E2E before every issue lands, and development is strictly serial (the decisive constraint / Decision 3) so there is no parallel-merge race for a per-PR E2E to catch. The per-PR CI E2E was therefore a redundant environment re-run, not first-line coverage.
>
> *New shape.* `ci.yml` is a **fast gate only** — build + unit + architecture (~1.5 min) on PRs and `main`. `release.yml` runs the **full suite incl. E2E + coverage** once in its `verify` job on a `v*` tag, then builds/publishes the semver images, then deploys to prod and smoke-checks `/alive`. The per-merge CD workflow (`cd.yml`) is **retired**; its deploy logic is folded into `release.yml`, keyed on the semver version instead of the commit SHA.
>
> *Guarantee delta.* "main cannot break" (see *Consequences*) softens to "main is **fast-green + locally pre-flighted**." The stronger guarantee is preserved and in fact moved *closer* to the deploy: **prod cannot ship a commit that fails the full CI suite**, because `verify` gates the release that gates the deploy. The residual exposure is a Docker-in-CI-only failure that the local pre-flight cannot reproduce — now surfaced at release time (blocking the release, before it reaches anyone's prod) rather than per-PR. Tracked in `plantry-49hm`.

**5. Release for self-host / OSS — tagged, semver, public.** Cutting a release is pushing a git tag, which builds and publishes **semver-tagged public images** plus a shipped Compose file and `.env.example` (see [self-hosting.md](../Operations/self-hosting.md)). **We proceed as if Plantry will be open-sourced** — the pipeline, image-visibility plan, and contributor surface are built OSS-ready — while the decision to actually make the repository public is **deferred and non-blocking**: the only OSS-gated work is flipping visibility and standing up the contributor surface (Phase 5 of the rollout). The runner policy follows from this assumption: public PRs run on **hosted** runners (self-hosted runners are unsafe for untrusted contributor code).

## Portability

| Piece | Portable? | Coupling / escape hatch |
|---|---|---|
| OCI image (Dockerfile) | Fully | Runs on any container host. |
| GHCR registry | Effectively | Only GitHub tie-in on the deploy side; moving = change the image prefix. |
| GitHub Actions YAML | Medium | Workflow *syntax* is GitHub-specific; the *steps* (docker build/push, ssh, compose) are plain shell and port to another CI in an afternoon. **This is the accepted, flagged tradeoff.** |
| Deploy = ssh + compose | Fully | Nothing GitHub-specific; works against any Linux host from any CI. |
| Compose / Postgres / reverse proxy | Fully | Vendor-neutral. |

The deliberate non-portable surface is the Actions YAML dialect alone; everything that *defines the deployment* stays generic. Abstracting the CI vendor away too (e.g. Dagger) is over-engineering at this stage. Leaving GitHub means rewriting ~1 workflow file and keeping everything else.

## Consequences

- **Divergence from ADR-012's "one composition for local and homelab."** The prod Compose file is separate from the AppHost. The Aspire-generated compose bundles the Aspire dashboard and passes the Postgres **superuser** password to the web app; production requires least-privilege (`app_user` only, ADR-017) plus a reverse proxy / TLS. The AppHost remains the single *local-development* app model; it is **not** the production runtime. ADR-012 is amended to record this.
- **CI must become green and at least as strong as the local pre-flight before it can gate.** Gating on a red or weaker CI is worse than not gating. The prerequisites (the SDK-version and workload bugs, and bringing E2E into CI) are tracked in [cicd-rollout-plan.md](../Operations/cicd-rollout-plan.md).
- **Workflow change.** The worker pushes branches and opens PRs instead of merging locally; the parallel `pipeline-orchestrator` skill describes a system that is not used and is retired or reduced to a serial driver. `bd close` moves to *after* the PR merges.
- **The loop's worst failure mode strengthens** from "idle, never a broken main" to "idle *and* `main` cannot break, because nothing merges without green CI." A new, milder stall mode appears: a PR stuck on failing CI (parked as `ci-failed`). *(Amended 2026-06-20 — see the amendment under Decision 4: with the slow suite moved to the release boundary, the guarantee on `main` is now "fast-green + locally pre-flighted"; the full-suite gate that "cannot break" protects is **prod**, at the release tag.)*

## Open

No blocking open decisions. The publish-as-OSS flip is deferred but non-blocking (Decision 5).

*Resolved 2026-06-19:* the full suite incl. E2E runs both locally (pre-filter) and in CI (authoritative) — Decision 4; a merge queue guards `main` against the concurrent-manual-push case — Decision 3; and the project proceeds as if OSS-bound — Decision 5.

*Amended 2026-06-20:* the merge queue (Decision 3) is **deferred until the repo is org-owned** — merge queues are unavailable on personal-account repos. Strict branch protection + the orchestrator mergeability guard cover the concurrent-manual-push case meanwhile. See the amendment under Decision 3.

*Amended 2026-06-20 (#2):* the slow suite (E2E + integration + coverage) moves from per-PR CI to the **release-tag `verify` gate**, and **deploy is release-gated** — `ci.yml` is a fast gate only and the per-merge `cd.yml` is retired (folded into `release.yml`). See the amendment under Decision 4. Tracked in `plantry-49hm`.
