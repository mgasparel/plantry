# ADR-009 — Binary content stored in PostgreSQL

**Status:** Accepted · 2026-06-06

## Context

The app holds binaries: receipt images (intake) and recipe photos. The usual options are object storage (S3/MinIO), the local filesystem, or the database itself.

## Decision

Store binary content **in PostgreSQL** (`bytea`, or large objects if individual items grow large), alongside the rest of the system of record.

## Consequences

- **One backup story.** A single `pg_dump`/PITR stream captures data and binaries together — no second system to back up, sync, or get out of step with the database. This is the deciding factor at current scale (single household, modest image volume).
- Transactional consistency: an image and the record that references it commit or roll back together.
- Accepted cost: large blobs add row/WAL bloat and DB size; binaries don't get CDN/object-store ergonomics. If volume ever makes this hurt, the access is already behind a repository, so swapping to object storage is a contained change — revisit then, not now.
