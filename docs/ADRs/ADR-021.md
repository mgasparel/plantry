# ADR-021 — Read paths may cross bounded-context boundaries via read-only cross-schema read models

**Status:** Accepted · 2026-06-27

## Context

ADR-010 makes Plantry a modular monolith: bounded contexts in one process and one Postgres database, each owning its aggregates in its **own schema** (`catalog`, `recipes`, `mealplanning`, `inventory`, `pricing`, …), referencing other contexts **by ID only**, reached only through anti-corruption **ports** (e.g. `IRecipeReadModel`, `ICatalogProductReader`, `IPriceReader`, `IInventoryStockReader`). ADR-006 already anticipates **read models carrying domain logic server-side** — "dedicated read-model endpoints for composed views (e.g. recipe fulfillment) — CQRS-lite." ADR-014 settled the *write* side: cross-context writes are eventual, never one shared transaction, because a shared transaction couples contexts at the persistence layer and that coupling cannot survive extracting a context.

This ADR settles the *read* side, forced by a concrete failure. The Meal Planner week page (`MealPlan/Index.cshtml.cs::LoadWeekAsync`) loads a week of planned meals and enriches each with live fulfillment % and cost. A single page load was traced at **6.09 s and ~4,453 spans**. The cause is structural, not a stray bug:

- **Each recipe is enriched up to three times per page** — `costingService.RollUpWeekAsync(plan)`, then per-meal `fulfillmentService.RollUpMealAsync`, then per-meal `costingService.RollUpMealAsync` — and the shared `GetEnrichmentAsync` computes *both* fulfillment and cost on every call, so each half runs three times.
- **Each enrichment fans out per ingredient across three contexts** — `catalogReader.FindAsync` per ingredient, `priceReader.FindLatestAsync` per ingredient, `unitConverter.ConvertAsync` per ingredient (twice) — each a separate round-trip through a per-aggregate port. Only the stock read (`FindStockBatchAsync`) is batched.
- **The RLS interceptor adds a `set_config` per connection open** (ADR-008), so spans run roughly 2× the logical query count.

The fan-out is `O(meals × dishes × ingredients × contexts)`. That is not a bug in the ports; it is what happens when a read-heavy projection is assembled through abstractions designed for the **write** side — one aggregate at a time, one context at a time. The boundaries that keep *commands* honest make this *query* pathological.

Two facts about our deployment make a different read-side answer both safe and cheap:

1. **One physical database, separate schemas.** A cross-context read is a fully-qualified cross-schema JOIN (`mealplanning.planned_dishes ⨝ recipes.recipe_ingredients ⨝ catalog.products`) — explicit about which boundary it reaches into, and physically free.
2. **Tenancy is enforced at the connection, not the context.** `HouseholdRlsConnectionInterceptor` arms `set_config('app.household_id', …)` on every connection open, and each table's Postgres RLS policy keys off that GUC. A raw cross-schema query run on an RLS-armed connection is therefore household-isolated for **every** table it touches, regardless of owning context — the database filters, not the per-context EF query filter.

## Decision

**A read path may use a dedicated, read-only read model that queries across context schemas directly, where doing so collapses an `O(n)` cross-context fan-out into a small, flat set of queries.** Such a read model is governed by four rules:

1. **SQL fetches data; C# keeps the math.** The cross-schema query loads raw inputs in bulk (planned meals, dishes, recipe ingredients, product facts, stock snapshots, latest prices, units, conversions). Fulfillment %, cost roll-up, expiry classification, variant rollup, and unit conversion stay in the existing domain services, which become **pure functions over already-loaded data** rather than services that issue per-item queries. Domain logic never moves into SQL. This is the line that actually protects DDD; everything else is plumbing.
2. **Read-only and one-directional.** The read model never writes. Only the consuming surface depends on the producing contexts' tables — never the reverse. Writes and invariants still go through aggregates and ports, unchanged (ADR-010, ADR-014).
3. **Lives in the composition root.** The read model lives in `Plantry.Web`, the one project that already legitimately references every context. The context projects keep their `→ SharedKernel only` dependency; no context gains knowledge of another.
4. **Guarded by tests, not by indirection.** Because the query hard-codes other contexts' physical columns, a contract integration test (query against the real migrated schema) and an RLS isolation test (two households, one read model, no leakage) replace the compile-time safety the port used to give. Those tests *are* the anti-corruption layer for this seam.

**Flatness is the goal, not merely fewer queries.** The target is a query count independent of how many meals, dishes, or ingredients are on the page — single-digit to low-teens, whether the week holds 3 meals or 30.

### Rejected alternatives

**Per-recipe batching of the existing ports only.** Batching the catalog/price reads inside a single `ComputeAsync` removes the per-ingredient fan-out but leaves chattiness `O(distinct recipes × contexts)` — "dozens," not flat. A worthwhile cheap win, and a legitimate boundary-pure fallback, but on its own it does not reach *sane*. (Page-wide batching purely through ports — gather every id across the week, one batched call per context, stitch in memory — *is* co-equal to the JOIN on query count and is strictly boundary-purer. We permit the cross-schema JOIN over it only where a relational walk like recipe→ingredient→product is genuinely simpler as a JOIN than as batch-load-and-stitch. Where stitching would otherwise explode rows — stock lots, price history — we keep separate batched queries, not one mega-JOIN.)

**Redis cache / data replication / event-carried state transfer.** These are the right answers to a *network* boundary — they exist to avoid a remote call. We have one process and one database; an in-process request-scoped memo is the entire "cache," and a SQL JOIN is the entire "composition." Adding a cache server or a replication projection here pays distributed-systems costs to solve a problem we do not have. Deferred until a context is actually extracted.

**Pushing fulfillment/cost math into SQL.** Tempting for a true single-query page, but it relocates domain rules into the query layer, creating two sources of truth for "is this cookable / what does it cost." This is the actual DDD violation the ADR exists to forbid — see rule 1.

**A shared transaction across contexts (the write-side analogue ADR-014 rejected).** Not applicable — the read model is read-only, so there is no transaction to share and no consistency coupling to create. This is precisely why the read-side answer can be more permissive than the write-side one (below).

## Why this is consistent with ADR-014 rejecting cross-context coupling

ADR-014 refused a shared *connection/transaction* for writes because it creates a **consistency** dependency — two contexts' state changes bound into one ACID unit — that cannot survive a split and converts the monolith into a distributed monolith. A read-only cross-schema JOIN creates a **schema-shape** dependency only: the read side knows another context's columns. That dependency is (a) one-directional, (b) incapable of corrupting state, and (c) **recoverable on the same terms ADR-014 set for writes** — if a context is later extracted, the read model degrades to API composition (batch calls + in-memory join), the *same shape* the SQL JOIN expresses, exactly as ADR-014's saga "shape is identical in-process and across a network." The join simply moves from the database to memory. We are not reintroducing the coupling ADR-014 forbade; we are taking, on the read side, the read-side version of the same splittability bargain.

## Consequences

- **Splittability is preserved (ADR-010 honoured).** No context depends on another's transaction or internals. The post-extraction fallback (API composition) is named in advance and is the natural refactor of a read model that already gathers ids and composes results.
- **Tenancy rests entirely on connection-level RLS for these paths.** The read model bypasses per-context EF `HasQueryFilter`, so the Postgres RLS policies (ADR-008) are the sole backstop — which is correct, they always were. The mandatory RLS isolation test makes a silent regression impossible.
- **A new failure mode: schema drift breaks reads at runtime.** Renaming `catalog.products.track_stock` no longer fails to compile; it fails the contract test in CI. We accept one runtime-shaped dependency (caught by a test) in exchange for deleting an `O(n)` cross-context fan-out. For a hot read path, a good trade; for a cold one, prefer the ports.
- **Scope discipline.** This sanctions cross-schema reads only for **read models on demonstrated hot paths**, not as a general license to JOIN. The default for reads remains the ports; a read model is a deliberate, tested exception with a recorded justification (a trace).
- **Domain logic stays put.** Because the math remains in the domain services, the write path, the AI planner, and the read model all compute fulfillment/cost identically — no second implementation drifts in SQL.

## Relationship to other decisions

The read-side dual of **ADR-014** (which governs cross-context *writes*): both keep contexts splittable, ADR-014 by refusing shared write transactions, this by confining cross-schema coupling to a read-only, test-guarded query model with a known post-split fallback. Realises **ADR-006**'s "dedicated read models for composed views (e.g. recipe fulfillment), CQRS-lite," and specifies *how* such a read model may cross schemas. Relies on **ADR-008**'s connection-level RLS for tenant isolation. Operates within **ADR-010**'s modular monolith and its splittability promise. First applied to the Meal Planner week page; tracked in the `bd` issues opened with this ADR.
