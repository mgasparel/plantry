# ADR-022 — AI-suggested conversions are provisional deterministic values, tagged in-place, not a quarantined subsystem

**Status:** Accepted · 2026-07-04

## Context

`product_conversion` (catalog.md DM-12) carries a per-product, cross-dimension or density factor — `1 cup flour = 120 g`, `1 egg = 50 g` — that `UnitConverter` bridges through so cross-dimension math (lb → each, cup → g) resolves. catalog.md draws a **dividing line for conversions**: anything *universal* (`kg = 1000 g`, `dozen = 12`) is a `factor_to_base` value on `unit`; anything that *depends on the product* is a `product_conversion` row. Both sides of that line are **deterministic** — a single stored number the converter trusts.

A new need breaks the implicit assumption that every product conversion is user-authored. Intake wants to *learn* a conversion — an LLM reading a receipt can infer "1 lb bananas ≈ 5 each" — and have that factor be **usable for computation immediately** (so weight→each intake resolves) while still being clearly **unconfirmed**, with a one-click path for the user to endorse it. plantry-1mu (weight→each conversion during intake) is the first consumer, but the user's intent is explicitly broader: the same *suggested → confirmed* provenance pattern is meant to be reused elsewhere in the app.

The tension: an AI-guessed factor must flow through the exact same converter as a hand-entered one (cross-dimension math must "just work"), yet the user must never be surprised that "random conversions appeared out of nowhere," and must be able to promote a guess to authoritative.

This sits against **ADR-011 / Gate 5** — "AI as untrusted external function," which quarantines raw model output in a *staging aggregate* (`ImportSession`/`ImportLine`, `MealPlanProposal`) until an explicit user confirmation commits it into a core aggregate. The question this ADR settles is whether a learned conversion factor is such raw output (and must be quarantined) or something else.

## Decision

**An AI-suggested conversion is a *provisional deterministic value* that lives in the same `product_conversion` system as a user-confirmed one, tagged with its provenance — not a separate approximation subsystem and not quarantined out of the converter.**

Concretely:

1. **One source of truth, tagged.** `product_conversion` gains a `source` provenance scalar (`ConversionSource`: `user_confirmed | ai_suggested`). There is **no separate "suggested conversions" table.** The converter reads one collection; provenance is metadata carried alongside the factor, never a second store to reconcile.

2. **Provenance is metadata, not a computation gate.** `UnitConverter.Convert` and `ReachableUnits` treat an `ai_suggested` factor **identically** to a `user_confirmed` one — an `ai_suggested` lb→each bridge is usable the moment it exists. The converter stays pure and provenance-blind; nothing in the resolution order (same-unit → same-dimension → product conversion → fail-loud) inspects `source`.

3. **Lifecycle: suggested → confirmed, through the aggregate root.** `Product.AddConversion` gains a `source` parameter defaulting to `UserConfirmed` (existing callers unchanged). `Product.PromoteConversion(conversionId)` flips `ai_suggested → user_confirmed` and is **idempotent** — promoting an already-confirmed conversion is a no-op success. Children are still created and mutated only through the `Product` root (ADR-010 aggregate boundary).

4. **Merge rule for a colliding `(fromUnit, toUnit)` pair.** A user-confirmed factor **supersedes** an AI-suggested one for the same ordered unit pair: adding a confirmed conversion drops any suggested conversion for that pair so the two never fight. An AI suggestion never overwrites or duplicates an existing conversion (of any provenance) for the same pair — the seed is idempotent and returns the existing row. (Two *user-confirmed* entries for the same pair are left as-is; this ADR does not change historical confirmed-vs-confirmed behaviour.) A manual edit of a factor's value is, by design, user authorship and should land as `user_confirmed` (there is no edit-conversion path today; when one is added it must apply this rule).

5. **Surfaced in the UI, hypermedia-only.** Wherever product conversions are listed/edited (the Catalog product detail page), an `ai_suggested` conversion is visibly tagged and carries a **Confirm** (promote) action — a plain htmx/Razor form POST, no island, no client-side domain logic (ADR-020 default).

6. **Backfill.** Existing rows migrate to `user_confirmed` (they were all implicitly user-authoritative). The column persists as `text` + `CHECK (source IN ('user_confirmed','ai_suggested'))` (Gate 7), household-scoping and RLS unchanged — `source` is a scalar on the already household-scoped row.

## Why this does not violate ADR-011 / Gate 5 ("AI as untrusted external function")

Gate 5 quarantines **raw, unreviewed model output** from writing into a **core aggregate** without an explicit user review step. That is about *provenance of a write* into stock/catalog/pricing state — a receipt line the model parsed must sit in `ImportSession` until the user confirms it, because committing it changes what the household *owns*.

A learned conversion factor is a different kind of object. It does not assert that any stock exists, was consumed, or was priced; it asserts a **relationship between two units for one product** — a piece of reference math, not a state change. Making it computationally live changes no inventory, no ledger, no money. The two safeguards Gate 5 actually cares about still hold:

- **The seeding of a suggested conversion is itself a staged, user-confirmed act.** In the first consumer (plantry-1mu), the LLM's guess is proposed during Intake review; the user's confirmation of that import is what writes the `ai_suggested` row. The raw model output is still staged in `ImportLine`/`raw_parse` per Gate 5; what lands in catalog is a typed, resolved factor — exactly Gate 5's "only user-resolved, typed fields commit."
- **The `ai_suggested` tag *is* the visible, promotable review surface.** Rather than hide a guess in a quarantine table where Recipes and Pricing can't see it, we keep it in the one converter every context reads and mark it unconfirmed. The user reviews it in place and promotes it. This is *more* transparent than a sibling table, not less.

A rejected sibling-table design would have duplicated `UnitConverter`'s reachability/bridging logic across "confirmed" and "suggested" stores and hidden suggestions from Recipes/Pricing until promotion — creating two sources of truth for the same math (the exact failure ADR-021 rule 1 and ADR-011's "single source" instinct warn against). Tagging in place keeps one converter, one store, one behaviour.

## Consequences

- **The converter stays pure and provenance-blind.** No branch on `source` in resolution; a suggested bridge resolves cross-dimension math immediately. This is the whole point — plantry-1mu's weight→each math works the moment the guess is seeded.
- **Provenance is a reusable primitive.** `ConversionSource` + the `AddConversion(source)` / `PromoteConversion` lifecycle is deliberately general. The user intends to reuse the *suggested → confirmed* shape elsewhere; future uses should follow the same "tag in place, promote through the root, converter/consumer stays provenance-blind unless it has a reason not to" pattern rather than minting per-feature quarantine tables.
- **A suggested factor can be wrong and is still live.** Accepted deliberately: a usable-but-unconfirmed factor beats a blocked flow (Plantry's friction thesis, Gate 8). The tag communicates uncertainty, the merge rule prevents a stale guess from outliving a real answer, and promotion/removal give the user control. A consumer that needs to *withhold* action on unconfirmed factors (none today) can read `source` — it is available, just not consulted by default.
- **Deterministic dividing line, refined not redrawn.** catalog.md's universal-vs-product-specific line is unchanged. This ADR adds an *orthogonal* axis on the product-specific side — confirmed vs suggested — where a suggestion is a *provisional* value on the deterministic side of the line, not a new "approximation" concept. When promoted it is indistinguishable from any other product conversion.
- **Tests guard the seam.** Domain (add-with-source, idempotent promote, merge/supersede, aggregate-boundary), converter (resolves an `ai_suggested` bridge), and migration (existing rows backfill to `user_confirmed`) are covered.

## Relationship to other decisions

Refines **ADR-011 / Gate 5** by distinguishing a *staged state-changing write* (quarantined) from a *provisional reference value* (tagged in place, computationally live). Operates within **ADR-010**'s aggregate boundary — the child mutates only through `Product`. Consistent with **ADR-021**'s "one source of truth for the math" instinct (rejecting a sibling table that would fork converter logic). UI follows **ADR-020**'s hypermedia default (the Confirm action is a plain form POST, not an island). First consumer: plantry-1mu (weight→each conversion during intake); the provenance pattern is intended for reuse beyond conversions.
