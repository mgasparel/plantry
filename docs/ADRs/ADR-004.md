# ADR-004 — Server-rendered hypermedia, not a SPA

**Status:** Accepted · 2026-06-06

## Context

The early lean was a TypeScript SPA framework (Svelte / Next.js / React). On inspection that fits Plantry poorly:

- The app is private (single household, auth-gated) with no public/SEO surface — SSR/SSG buys nothing.
- "Lots of data" is **domain state** (just fetching), not **client interaction state**. Genuine client-held interaction state is concentrated in a few spots (import review form, cook-with-substitutions, meal-plan review). Most pages are read-then-act, the server-render sweet spot.
- A framework with a server tier (esp. Next.js) creates **two backends** and constant "does this go in .NET or here?" pressure — a direct violation of ADR-001.

## Decision

Server-render pages from .NET (Razor Pages/MVC). Drive interactivity with hypermedia: the server returns HTML fragments. No SPA, no Next.js, no Blazor.

## Consequences

- No serialization boundary: the domain renders its own state as HTML instead of projecting aggregates into a JSON contract the client re-models. This is the most DDD-faithful UI approach — there is no client-side domain to drift out of sync.
- No public API surface by default. The "API" is C# calls into application services (see ADR-006 for the conditional case).
- Cost: rich interactions push work onto the server (many partial-view endpoints, form-field indexing). Accepted trade.
- Blazor rejected: Blazor Server ties every interaction to a live SignalR circuit; Blazor WASM is a full SPA payload. Both inherit SPA trade-offs without the ecosystem.
