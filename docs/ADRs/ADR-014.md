# ADR-014 — Cross-context writes are eventually consistent, never one shared transaction

**Status:** Accepted · 2026-06-14

## Context

ADR-010 establishes the modular monolith: nine bounded contexts in one process and one database, each owning its aggregates, referencing others **by ID only**, and modified **one aggregate per transaction**. Its promise is that "splitting one out later, if ever needed, is contained because the boundary already exists." ADR-011 makes Inventory's `Consume` the single primitive every pantry decrement flows through.

P2-3 (Cook a recipe) is the first operation that *writes across two contexts in one user action*: it records a `CookEvent` (Recipes) **and** decrements stock via `Consume` (Inventory). The P2-3 plan and the implementing epic (`plantry-so3`) specified that the `CookEvent` and its consumes be written in **one transaction**. Implementation review (`plantry-292`) found this was both unmet *and* the wrong target:

- The two writes already land on **separate DbContexts on separate physical connections** (`RecipesDbContext`, `InventoryDbContext`) — same Postgres instance, but no shared transaction. `ConsumeStockCommand` opens and commits its **own** transaction per consume.
- Making them genuinely atomic requires either a **shared connection + local transaction** spanning both contexts, or a **distributed transaction** (`TransactionScope`/2PC).

This forced the strategic question the cook flow only surfaced: **when an operation spans bounded contexts, what is the consistency model?**

## Decision

**Consistency across bounded-context boundaries is eventual. We never wrap two contexts' aggregates in one ACID transaction.** Each aggregate is written in its own transaction (upholding ADR-010's one-aggregate-per-transaction rule literally, across contexts as well as within them). A cross-context operation is modeled as a small **saga**: one aggregate is the authoritative anchor, and the other context's write follows as an idempotent, recoverable step.

**Rejected — shared connection + single local transaction.** It works *only* because the contexts currently share a database. It couples them at the persistence layer — the one coupling that cannot survive extracting a context into its own service — converting ADR-010's "splittable later" monolith into a distributed monolith the day a split is attempted. It is cheap to add and expensive to remove, and it buys atomicity for a consistency need that does not require it (below). The aggregate-as-consistency-boundary rule (ADR-010 §Aggregates) exists precisely to keep this seam clean; a cross-context transaction is the rule's exact violation.

**Rejected — distributed transaction (2PC / `TransactionScope`).** Two connections escalate to MSDTC, which is unavailable on our Linux/Docker runtime (ADR-012) and is the heavyweight wrong tool for a single database. Universally avoided.

**The deciding test — invariant vs. drift.** Choose a shared transaction *only* when a true business **invariant** demands atomicity (one fact must never exist without the other) **and** the boundary is one we have committed never to split. Absent both, eventual consistency is not a compromise — it is the correct model. Cooking has no such invariant: a `CookEvent` recorded a few hundred milliseconds (or one retry) before its stock decrement harms nothing — no money, no double-spend, no safety property. So the epic's "one transaction" acceptance criterion was **over-specified**; it described ACID atomicity for what is a cross-context saga.

## Mechanism — the cook, as the worked example

The pattern every future cross-context write should follow:

1. **Anchor the truth in one aggregate, written first.** `CookRecipe` writes the `CookEvent` in Recipes' own transaction *before* issuing any consume. The cook is then a durable fact; the consumes are a follow-on. *(The original implementation had this backwards — consumes committed first, so a mid-flow crash could decrement stock with no cook record at all, the strictly worse direction.)*
2. **Make the downstream write idempotent.** Each consume is keyed by `cookEventId` (+ line), so a retry cannot double-decrement.
3. **Recover from durable state, not a separate ledger.** Every Inventory journal row a cook writes is already stamped `sourceRef = cookEventId` (ADR-011). Drift is therefore **detectable from data we already persist**: a `CookEvent` lacking its expected journal rows is an unfinished cook, closed by a reconciliation/retry keyed on `cookEventId`. The anchor aggregate *is* the durable intent.

Crucially, **this needs no transactional outbox** (see `plantry-3n7`). A generic outbox is the right tool when the downstream effect leaves no trace to reconcile against; here the `CookEvent` + `sourceRef`-stamped journal give us that trace for free. An outbox remains deferred until a cross-context reaction appears that *cannot* be reconciled from its own durable output.

## Consequences

- **Splittability is preserved (ADR-010 honoured).** No context depends on another's transaction. Extracting Inventory later changes only the transport of the consume command, not the consistency model — the saga shape is identical in-process and across a network.
- **A bounded inconsistency window is accepted by design.** Between anchor-write and downstream completion, the two contexts can disagree. This is permitted *only* where no invariant forbids it; the reconciliation path bounds how long it can persist. Any future cross-context write must make the same invariant-vs-drift call explicitly — and if it finds a real invariant, the answer is to **redraw the aggregate boundary so the invariant lives inside one aggregate**, never to introduce a shared transaction.
- **Cost moves from infrastructure to the flow.** We pay for optionality with idempotency keys, ordered writes, and reconciliation in each saga — not with distributed-transaction machinery. This is the intended trade.
- **Read models already tolerate it.** Fulfillment % and cost/serving are query-time read models (ADR-010 §Aggregates), so a transient post-cook lag surfaces as a momentarily stale derived view, self-correcting on the next read — not as corrupt persisted state.

## Relationship to other decisions

Extends **ADR-010** (makes one-aggregate-per-transaction explicit *across* contexts and names the saga as the integration pattern for cross-context writes) and **ADR-011** (relies on the `sourceRef` stamp as the reconciliation key). First applied in P2-3 via `plantry-292`; the deferred generic outbox is `plantry-3n7`.
