# ADR-005 — htmx + Alpine, no front-end framework, no Node toolchain

**Status:** Accepted · 2026-06-06 · Amended 2026-06-22 (escape hatch triggered; scope narrowed to the default — see [ADR-020](ADR-020.md))

## Context

Three developer-weighted concerns push against front-end frameworks, in order of weight:

1. **Dependencies (strongest).** npm's deep transitive trees vs. NuGet's shallower, curated ones — supply-chain surface and churn the developer does not want to own.
2. **Network payload.** Mobile-first; React+react-dom is ~45KB gzipped before app code (real apps 150KB+).
3. **Build time.** Less about Vite's speed (it's fast) and more about avoiding a *second build pipeline* alongside .NET.

An "islands" compromise (htmx everywhere + one framework component for the hard modals) was considered and rejected **for this value function**: an island reintroduces the npm tree and second pipeline (concerns #1 and #3) to win on payload (#2) — the one axis htmx already wins. Bad trade.

## Decision

- **htmx** for all server-driven interaction and HTML-fragment swaps (active search, cascading dependent selects, add/remove rows). Single ~14KB file, no package manager.
- **Alpine** for client-side local and draft state, including draft-collection modals — `x-data` array + `x-for` rows + `fetch` for catalog/unit reads. Another ~15KB file.
- **No Node in the repo.** Both libraries served from wwwroot or CDN.

## Consequences

- All three concerns satisfied: zero dependency tree, minimal cached payload, one toolchain (`dotnet watch`, no bundler).
- Reinforces ADR-001/004: server stays authoritative; product units, conversions, and fulfillment are domain reads rendered by the server.
- The recurring composite widget — *catalog search + cascading unit load + quantity* — appears across manual add, inline create, import review, recipe create/edit, cook substitution, and shopping add. It is reused as a shared server partial + a units endpoint rather than a packaged component.
- **Escape hatch (contingency, not yet triggered):** if a single screen genuinely defeats Alpine, build that one widget as a **web component** (vanilla custom element, or Lit ~5KB usable without a build) before considering React/Svelte + Vite. Adopting a bundled framework would reopen ADR-005.
- Watch item (**resolved by [ADR-013](ADR-013.md)**): the **import review form** is the worst case. It was pressure-tested against Alpine and lands *within* htmx + Alpine — the escape hatch was **not** triggered. ADR-013 records the pattern that makes it work (server-authoritative aggregates delivered via targeted out-of-band swaps) and defines the bounded optimistic-store tripwire that would reopen this ADR if crossed.

## Amendment · 2026-06-22 — escape hatch triggered; this ADR narrows to the default ([ADR-020](ADR-020.md), bead plantry-2zvm)

The escape hatch above is now **triggered** — deliberately, and bounded — by [ADR-020](ADR-020.md). The "sprinkle of reactivity" premise this ADR was decided under did not hold for three stateful surfaces (Intake review, Meal Planner, Take Stock): `meal-editor.js` ended up hand-rolling a reactive runtime, and [ADR-013](ADR-013.md) needed three amendments plus a cross-feature test primitive to stop derived views drifting. Two of this ADR's three original concerns also moved — the build-pipeline objection (#3) was relaxed by the team, and payload (#2) cuts *against* a framework here, not for it, since the app ships ~50KB of JS today.

What this changes, precisely:

- **Scope narrows from "the whole app" to "the default."** htmx + Alpine remains the governing choice for every server-driven page (lists, CRUD, detail, search) — which is most of the app. This ADR is **not** superseded.
- **ADR-020 is authoritative on three enumerated stateful surfaces**, where a bounded client-side reactive layer replaces the hand-rolled Alpine reactivity. Adding a fourth surface is not a default — it must meet the same bar and be recorded as an ADR-020 amendment.
- **The strongest original concern (#1, the npm dependency tree) is preserved as a live constraint, not waived.** ADR-020 leaves the library open but names a *buildless vendored runtime* (a single ESM file, like Alpine is served today — no npm tree, no pipeline) as one branch of the open tech sub-decision; the "no dependency tree" value still has weight in that choice.
- The escape hatch's original phrasing ("build that one widget as a web component… before considering a bundled framework") is **relaxed**: the contingency is no longer "one widget, web component only." It is "a bounded reactive layer on enumerated surfaces, library deferred," under the ADR-020 domain-authority tripwire.
