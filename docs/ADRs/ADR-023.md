# ADR-023 — Amendment vs Correction: the ledger distinguishes "the record was wrong" from "reality drifted"

**Status:** Accepted · 2026-07-20

## Context

The stock journal is append-only (ADR-011): rows are never updated or deleted, and the sole
sanctioned fix-up was Take Stock with `reason = Correction`. Correction's semantics are physical —
recorded stock diverged from what is actually on the shelf, cause unknown or unrecorded. That made
it the only available tool for a categorically different problem: a **data-entry mistake** on the
original Purchase line (OCR misread, typo). Routing an entry mistake through Correction records a
falsehood — "1 lb purchased + 2 lb discovered on recount" instead of "3 lb purchased, entered
wrong" — polluting spend, unit-price, and shrinkage analytics.

The 2026-07-20 incident made the gap concrete: a mis-entered receipt quantity had no in-app fix and
was hand-corrected via raw SQL across `intake`, `inventory`, and `pricing` schemas, bypassing every
domain invariant. "Ask an admin to fix the DB" is not an acceptable escape hatch for a recurring
class of mistake that OCR-driven intake guarantees.

## Decision

- **`StockReason.Amendment`** joins the why-taxonomy (DM-14), reserved for "the original entry was
  wrong." `Correction` keeps its physical meaning. The two are never interchangeable.
- An amendment is an **additive compensating journal row on the original lot** (delta =
  corrected − effective, bidirectional), written only by a dedicated `ProductStock.AmendPurchase`
  method. The append-only guarantee holds verbatim; the Purchase row is never touched. True
  purchased quantity is `Σ(Purchase + Amendment)` per lot.
- **Eligibility is ledger-semantic, not clock-based**: the lot must be intake-sourced and active;
  the corrected quantity cannot fall below what the lot has already had consumed; and amendment
  **closes permanently once any `Correction` row exists for the product after the purchase** —
  a count has already absorbed the discrepancy, so a later amendment would double-count. The
  closure check is product-level because Take Stock's positive true-ups land as new lots and are
  invisible lot-locally.
- **`PriceObservation` follows with supersede-append**: the corrected observation is a new row
  (`AmendsId` → original, original gets `SupersededById` bound once, same one-time-bind precedent
  as `ResolveStore`); pricing reads filter to non-superseded rows. The immutable-price-event
  contract narrows to "never edited, possibly superseded."

Full design, eligibility mechanics, and UX decisions: `docs/DomainDesign/purchase-entry-amendment.md`
(origin `plantry-x3dy`).

## Consequences

- Waste/shrinkage analytics can finally trust `Correction`: entry mistakes no longer masquerade as
  physical drift. The same distinction unblocks the consume-side attribution work (`plantry-vmfe`).
- The journal's reason taxonomy now encodes *epistemic* cause (record-wrong vs reality-drifted),
  not just movement type — new fix-up flows must pick the honest reason, not the convenient one.
- Pricing read paths carry a standing `SupersededById IS NULL` filter; any new pricing query must
  apply it (or deliberately opt into the audit chain).
- The amendment path is deliberately narrow (committed quantity only). Wrong product/unit/price
  remain out of scope; extending amendment means extending this ADR's reasoning, not bypassing it.
