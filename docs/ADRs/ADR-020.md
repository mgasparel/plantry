# ADR-020 — Reactive islands for stateful surfaces (Intake, Meal Planner, Take Stock)

**Status:** Accepted · 2026-06-22 (bead plantry-2zvm) · Amended 2026-06-24 (test-time Node, bead plantry-2zvm.7) · triggers the ADR-005 escape hatch · amends [ADR-005](ADR-005.md), [ADR-013](ADR-013.md), [ADR-015](ADR-015.md)

## Context

[ADR-005](ADR-005.md) chose htmx + Alpine with no front-end framework and no Node toolchain, under an explicit value function weighing three concerns: dependency tree (strongest), network payload, and a second build pipeline. It assumed a *mostly server-rendered app with small amounts of reactivity sprinkled in*, and named a contingency: *"if a single screen genuinely defeats Alpine… adopting a bundled framework would reopen ADR-005."*

The sprinkle premise has been falsified by the code, not by opinion:

- **`meal-editor.js` hand-rolls a reactive runtime.** Manual `fetch` + `htmx.swap`, a debounced rollup POST, a DOM-walk over `_x_dataStack[0]` to reconnect out-of-tree dish-search results to component state, and hand-built `URLSearchParams` to dodge key-collapsing. Its header comments enumerate *three named bug classes* (`var`-token silent failure, `Object.fromEntries` collapsing keys, Alpine v2/v3 accessor regression) — every one an artifact of doing reactivity by hand.
- **The fragment-endpoint sprawl is structural.** MealPlan's code-behind is 1,233 lines across 17 handlers; Intake review is 626 lines plus a 561-line row partial.
- **[ADR-013](ADR-013.md) is the project repeatedly rediscovering one bug** — *a derived view drifts after a localized mutation* — and answering it by hand-building invariants (the five `*Oob` partials, `BuildProjections()`, the cross-feature `OobContract` test). Its own 2026-06-18 amendment records that a worker, a reviewer, and a three-pass critic all hit the identical drift on the meal planner and none found the existing decision.

The pain is **localized** to three stateful surfaces — **Intake review, Meal Planner, Take Stock** — where draft collections, derived aggregates, and out-of-tree interactions concentrate. The rest of the app (lists, CRUD, detail, search) is well served by hypermedia and gains nothing from a framework.

**Payload is explicitly not a driver.** The app ships ~50KB of JS today (htmx ~14KB + Alpine ~15KB + ~20KB hand-written component JS); `plenish.css` at 227KB dwarfs it. A reactive island must not bloat this — "don't bloat the response" replaces ADR-005's now-relaxed "no build step" constraint.

This ADR triggers ADR-005's escape hatch deliberately and bounds it.

## Decision

### 1. Bounded reactive islands on three enumerated surfaces

A client-side **reactive layer** (specific library deferred — see §6) is introduced for exactly three surfaces:

- **Intake review** (`Pages/Intake/Review`)
- **Meal Planner** (`Pages/MealPlan/Index` and its editor)
- **Take Stock** (`Pages/Pantry/TakeStock`)

Everything else stays **htmx + Alpine** (ADR-005 remains the default). New surfaces do **not** default to an island; adding a fourth requires meeting the same bar (concentrated draft/derived/out-of-tree state) and is recorded as an amendment here.

### 2. The rule — domain logic stays in the backend

The whole architectural decision is one line: **business/domain logic stays server-side; the island holds UI state only.** Concretely, the **island owns** ephemeral UI state — draft collections, open/closed drawers, selected-but-unsaved fields, and derived display values that are pure functions of state it already holds (counts, sums, progress, enable/disable). The **server owns every domain concern** — persistence, fulfillment %, cost/rollup, validation-as-truth, and any catalog-default or unit-semantics rule — exposed to the island behind JSON endpoints (reinforces [ADR-001](ADR-001.md) / [ADR-004](ADR-004.md)). §7 is the tripwire for crossing this line.

The per-surface data contract — exact hydration shape, endpoint names — is **implementation detail of each screen, not part of this decision** (§4 sketches Intake's, non-bindingly). The only thing that generalizes is the rule above, plus where the line falls when "is this domain logic?" is genuinely ambiguous — which is §3.

### 3. Where the line falls (the part the rule doesn't settle by itself)

"Domain logic stays server-side" is necessary but not self-applying: some logic is genuinely ambiguous between *domain* and *UI*. Reading the Intake code surfaced three such cases; they are adjudicated here so the classification is consistent (and surface-independent — the same line-drawing applies to Meal Planner and Take Stock), rather than relitigated per screen under load:

1. **A priority/derivation chain that decides initial form values stays server-side.** Intake's `ComputePrefill` (`user-resolved > receipt-parsed > product-default`) encodes domain-flavoured rules — *"receipt unit wins so 2 kg never silently becomes 2 each,"* *"expiry = today + DefaultDueDays."* It is "logic" that only fills a form, but it is **domain** logic → server. The island receives the *computed* values and renders them; it never re-derives the chain.
2. **Filling one field from reference data on user interaction is UI, not domain.** When the user re-selects a product, the island may fill empty unit/location/expiry from that product's hydrated defaults, including `today + dueDays` arithmetic. This is form-filling from data the island already holds → **UI**, island-allowed. The line: applying a single default is UI; owning the *priority chain* that chooses between competing sources (case 1) is domain. Case 2 may not grow into case 1.
3. **Validation is mirrored, not moved.** The island may mirror field guards (`quantity > 0`, unit/location required, new-product needs name+category) to disable Save and show inline hints — convenience. The server re-validates on every mutation and is **the truth**.

### 4. Illustrative seam (Intake — non-binding)

Sketch only, to make §2–§3 concrete; it documents one screen and binds nothing. **Server → island (hydration JSON on GET):** session header, reference data (`products` with defaults + `skus`, `units`, `locations`, `categories`), per line `{ line, prefill }` with prefill **computed** (case 1). Hydrate from this — no round-trip for first paint. **Island → server (each returns updated line state):** `saveLine` (the `LineEditInput` shape), `dismissLine`/`restoreLine`, `commit`→`/Intake/Done`, `discard`→`/Pantry`. **Derived → client `computed`:** chips, progress, commit bar, receipt total are pure functions of line `status` + `price ?? suggestedPrice` → cannot drift; the five `*Oob` partials, `BuildProjections()`, and `OobContract` retire here (see [ADR-013](ADR-013.md) amendment). Meal Planner and Take Stock have their own shapes; only §2–§3 transfer.

### 5. Islands are not a SPA

No client router; each island **mounts per page from server-rendered HTML, and dies on navigation**. The server owns routing, auth/session, and navigation. This does **not** reopen [ADR-004](ADR-004.md) (server-rendered hypermedia, not a SPA) — it is the bounded client-state exception ADR-004's reasoning anticipated, not a client-owned application shell.

### 6. Tech selection — Preact + htm + signals, buildless (decided 2026-06-22)

The reactive layer is **Preact + htm + `@preact/signals`, served buildless** as pinned ESM files from `wwwroot/js/islands/vendor/` (the same delivery model Alpine uses today). No npm tree, no bundler, no Node in the repo. This was chosen against three weighted values — a high-quality codebase legible to both humans and agents; no payload bloat; and avoiding npm's pain and supply-chain surface — in that priority order.

Why this and not the alternatives:

- **It satisfies all three values rather than maxing one.** Payload is ~6KB gzip (vendored). Supply-chain surface is three auditable files, not a transitive tree. And quality does **not** require a build here (next point).
- **Quality without a build.** Authoring is real components + `computed()` derived state. Type-checking and editor intelligence come from the editor's **built-in TypeScript** via `// @ts-check` + JSDoc + a `jsconfig.json` — *zero install*. For agents specifically, buildless means **the file that runs is the file they read** — no transpile, source maps, or build config to reason about, which is itself a legibility win a build removes.
- **Svelte (the better-DX option) was ruled out by the no-npm value**, since its DX is paid for precisely with the npm tree + a build. Its Svelte-5-runes recency also degrades agent legibility (workers regress to Svelte 4).
- **`computed()` is the exact primitive this ADR is built on** — the §4 "derived views that cannot drift" map 1:1 onto signals.

**The one accepted concession:** htm's tagged-template markup is type-checked more loosely than JSX would be (props/signals are fully typed; template HTML less so). Accepted for three bounded surfaces.

**No-rewrite escape hatch:** if that looseness bites, adopt **esbuild** (a single self-contained binary, ~no dependency tree) to compile JSX+TS with the *same Preact library* — a minimal auditable build, not an npm ecosystem, and not a framework change. This is the only sanctioned path to a build step, and adopting it is an amendment here, not a silent change.

The §7 tripwire and the payload constraint bind regardless.

### 7. The tripwire — domain logic crossing into the island reopens this ADR

If an island ever computes fulfillment, cost, validation-as-truth, or any catalog/unit-semantics rule client-side — anything beyond UI/draft state and pure-derived display values — that is the signal this decision was breached. It must be recorded as an amendment here, not absorbed silently. This inherits the spirit of ADR-013 §4's optimistic-store tripwire.

## Consequences

- **The drift machinery deletes on island surfaces.** What [ADR-013](ADR-013.md) built by hand (OOB bundle + projection builder + `OobContract`) becomes three `computed` lines. The invariant is enforced by the runtime — the *enforcing artifact* ADR-013's 2026-06-18 meta-lesson asked for — instead of by prose plus a test the next agent must remember to satisfy.
- **Two UI models in one repo is the standing tax.** The cost is legibility, not bytes or build: "is this a Razor+htmx page or an island?" must stay a bright, boring rule (§1 — three enumerated surfaces, no defaulting). This is the thing most likely to fray and is the one to police in review.
- **A new data-contract seam exists.** The server now emits JSON UI-state for three surfaces (server VM ↔ island props), a contract to keep in sync that does not exist under pure hypermedia. Bounded to the three surfaces.
- **Testing moves, roughly net-neutral.** `OobContract`-style assertions retire on island surfaces; the islands need a client-side test story or lean on the existing Playwright e2e. The domain stays server-tested as today.
- **Free responsiveness wins fall out.** E.g. the Intake product picker's per-keystroke server round-trip (`OnGetFilterProductsAsync`) becomes instant client-side filtering over the already-hydrated `products` list — no boundary concern, since it filters a list the island already holds.

## Relationship to other ADRs

- **[ADR-005](ADR-005.md) — amended (scope-narrowed).** Its escape hatch is triggered for the first time. htmx + Alpine remains the default for server-driven pages; ADR-020 is authoritative on the three enumerated stateful surfaces. ADR-005 is *not* superseded — it still governs most of the app.
- **[ADR-013](ADR-013.md) — amended (superseded-in-practice on island surfaces).** The OOB-bundle contract is retired on Intake and the MealPlan rail; it remains the governing pattern for any htmx surface that grows a derived view. Its meta-lesson motivates this ADR.
- **[ADR-015](ADR-015.md) — amended (new third category).** Island data endpoints are UI-state JSON for a reactive client consumer — distinct from both ops-JSON and hypermedia-UI handlers.
- **[ADR-001](ADR-001.md) / [ADR-004](ADR-004.md) / [ADR-006](ADR-006.md) — reinforced, unchanged.** Domain stays server-authoritative; islands are not a SPA; the JSON endpoints are application-service-backed reads/commands.

## Amendment · 2026-06-24 — front-end test runner: Node permitted, test-time only (bead plantry-2zvm.7)

§6 said "no Node in the repo." That was overbroad: it conflated *Node in the shipped runtime/build* (which we still reject) with *Node as a dev test tool* (which we now need). The islands hold pure transforms — hydration→signal-state factories (`makeLine`/`makeRow`), draft→POST-body builders (the `LineEditInput`/`SaveRequest` shapes that **mutate real inventory**), and display `computed`s — that currently have **zero unit coverage**. The Consequences above already named this gap ("the islands need a client-side test story"); the seam-hardening work (bead plantry-eoj5) sharpened it by showing the contract test proves the JSON *shape* but nothing about what the island *does* with it. Owner decision (2026-06-24): front-end code must be testable, full stop.

**Decision.** Node is sanctioned as a **test-time-only** dependency. The rig is Node's **built-in test runner** (`node --test`) — zero npm dependency tree, no bundler — importing the ESM island modules directly. (Node 25.2 / npm 11.6 confirmed in the dev environment.)

**What this narrows (§6).** "No Node in the repo" → **"no Node in the shipped runtime or the build of shipped JS."** The browser path stays buildless and unchanged: no bundler, no transpile of shipped JS, no npm dependency tree / `node_modules` in what ships, no import map. *The file that runs is still the file you read.* esbuild remains the separate, still-unexercised escape hatch for transforming shipped code (§6) — this amendment does **not** trigger it.

**Boundaries that bind.**
- Node / `node_modules` must never enter the shipped path. A `package.json`, if added, is a **dev/test manifest only** (a `test` script + `type: module`), declares **no dependencies**, and `node_modules` is gitignored.
- The §7 domain tripwire is unaffected — testability is not licence to move domain logic into an island; it makes the *UI/draft transforms* that legitimately live there verifiable.
- Consequence for review: "no JS test rig exists" is no longer a valid reason to defer island-logic coverage once the rig lands (review-criteria Gate 6 updated to match).

Standing up the rig and writing the transform tests are tracked as separate implementation beads under epic plantry-2zvm.
