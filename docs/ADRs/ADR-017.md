# ADR-017 — Database migrations applied by an explicit one-shot migrator, not on app startup

**Status:** Accepted · 2026-06-19

## Context

`Plantry.Web` currently applies EF Core migrations at startup: `Program.cs` calls `Database.MigrateAsync()` for all eight `DbContext`s on every boot, in **every** environment, using the database **owner** connection (the migrations create schemas, RLS policies, and the `app_user` role). At runtime the app instead connects as the least-privilege `app_user` role so the Postgres RLS policies actually apply — RLS, even `FORCE`, never restricts a superuser/owner.

Migrate-on-startup has three problems, in increasing order of how much they matter here:

- **It races across instances** — two booting instances both attempt DDL. Latent today (one instance), but a one-way door.
- **It crash-loops on a bad migration** instead of failing as a discrete, readable deploy step.
- **It forces the web process to hold owner credentials for its entire life** just to run a deploy-time check — which undermines the very least-privilege split the RLS design depends on. The runtime path deliberately connects as `app_user`; startup migration keeps owner creds in the same process.

This decision also resolves a **discovered bug**. The `app_user` password is hard-coded in the initial Identity migration:

```
InitialIdentitySchema.cs
CREATE ROLE app_user LOGIN PASSWORD 'app_user_password' NOSUPERUSER NOBYPASSRLS;
```

…while `Program.cs` **requires** `Database:AppUserPassword` from configuration in production (throws if unset) and connects as `app_user` with **that** value. The two never reconcile: if an operator sets the password to anything other than the literal `app_user_password`, the role's actual password is still the hard-coded one and the app cannot authenticate to its own database. Self-hosting makes this untenable — every instance would ship the same public credential, and the operator could not change it.

## Decision

**Extract migration into a dedicated one-shot `Plantry.Migrator` console project.** It:

1. Runs the same eight `MigrateAsync()` calls with the **owner** connection (reusing the proven loop currently in `Program.cs`).
2. **Reconciles the `app_user` role password from configuration** — `ALTER ROLE app_user WITH PASSWORD <Database:AppUserPassword>` (creating the role first if absent) — so the running app's configured credential always matches the role.
3. Exits.

`Plantry.Web` no longer migrates on startup (the block is removed, or gated to `Development` so local F5 stays zero-friction).

**The migrator runs as a one-shot container.** It is a service in the Compose stack gated by `depends_on: { migrator: { condition: service_completed_successfully } }` on the web service. In your own CD it is additionally invoked explicitly first (`docker compose run --rm migrator`) so a migration failure surfaces cleanly in the pipeline logs; self-hosters get it automatically through Compose ordering on `docker compose up`. **Owner credentials go only to the migrator; the web container receives `app_user` credentials only.**

Migration *bundles* (`dotnet ef migrations bundle`) were considered and rejected: with eight migration assemblies across eight contexts, one looping migrator that reuses the existing code is simpler than eight bundles.

## Consequences

- **Least privilege is restored.** A compromised web container cannot create roles or alter the schema — it holds no owner credentials.
- **The `app_user` password becomes per-instance, operator-controlled, and rotatable** — change `.env`, re-run the migrator. The hard-coded literal in the migration degrades to a bootstrap default that is immediately overwritten.
- **One migration mechanism serves both deployment targets** — your prod CD and third-party self-host both run the same migrator image the same way (see [deployment.md](../Operations/deployment.md), [self-hosting.md](../Operations/self-hosting.md)).
- **Migration discipline, made explicit for multi-version self-host upgrades:** `MigrateAsync` applies all pending migrations in order, so a self-hoster jumping several versions is upgraded correctly in one shot — *provided* migrations stay **forward-only**, old migrations are **never squashed or deleted**, and destructive changes get a deprecation window (drop a column a release after writes to it stop). Down-migrations exist but downgrading a live database is unsupported in practice; the contract is **forward-fix + back-up-before-upgrade**.
