# ADR-015 — Operational/maintenance endpoints may return JSON; UI handlers stay hypermedia

**Status:** Accepted · 2026-06-15

## Context

ADR-004 makes Plantry's UI server-rendered hypermedia: handlers return HTML (full pages, `Partial` fragments, `Content`, or `RedirectToPage`), never a JSON contract a client re-models. ADR-006 reasons through the *conditional* case of a deliberate HTTP/MCP API over application services, should one ever be needed.

The `ReconcilePendingCooks` work (plantry-292c) added an explicit on-demand trigger — `POST /Recipes/ReconcilePending` — as the manual/automation counterpart to the opportunistic sweep that runs at every `CookRecipe` entry. It returns `{ reconciledLineCount: N }` as JSON, which is the **only** `JsonResult` in the `Pages/` tree. A pre-flight review flagged it: the rule for whether an endpoint like this may return JSON was undecided and undocumented (plantry-8fy).

The endpoint is genuinely machine-facing. It renders no page (its `.cshtml` is body-less), has no interactive UI consumer, and exists to be driven by a human running a one-off command, a maintenance script, or a future scheduled job. Forcing it to emit an HTML fragment would produce markup nothing displays and that a script must then parse — hypermedia ceremony with no hypermedia consumer.

## Decision

Endpoint response shape follows the **consumer**, not a blanket rule:

- **Interactive, human-facing handlers** — anything a browser navigates to or an htmx interaction targets — return **hypermedia** (ADR-004): a page, a `Partial` fragment, `Content` HTML, or a redirect. This remains the default and covers essentially all of the app.
- **Operational/maintenance endpoints with no interactive UI** — reconciliation, health/diagnostics, and similar backstops invoked by a human command, a script, or a scheduled job — **may** return JSON.

`POST /Recipes/ReconcilePending` is the canonical example and stays as-is. The qualifying test is strict: the endpoint must have **no** interactive UI surface that consumes it. The moment a screen, button, or htmx interaction drives an endpoint, it is human-facing and returns hypermedia — a JSON response that some Razor view then has to consume in JavaScript is exactly the serialization boundary ADR-004 exists to avoid, and is out of scope here.

This is narrower than ADR-006: ADR-006 governs a *deliberate, designed* API surface (REST around aggregates, read models, MCP). This ADR covers incidental single-purpose operational hooks that are not part of any such API. If these hooks ever grow into a real maintenance API, ADR-006's REST-around-aggregates shape governs it.

## Consequences

- The lone `JsonResult` outlier becomes an intentional, bounded exception with a written boundary, rather than an accident that silently licenses JSON sprawl across the UI. New operational endpoints have a precedent to point at; new *UI* endpoints have no cover for returning JSON.
- The boundary is a bright line — "is there an interactive UI consumer?" — so the FIX/keep call on any future handler is mechanical, not a matter of taste.
- These endpoints carry no client-held domain state and project no aggregate into a contract, so ADR-004's core concern (a client-side domain drifting out of sync) does not arise: the payload is a flat operational result (`{ reconciledLineCount }`), not a re-modelled aggregate.
- If an operational hook later needs to be driven from the UI (e.g. a "Reconcile now" button with a result toast), it converts to a hypermedia handler returning a notice partial at that point — the JSON form was only ever justified by the absence of such a consumer.
