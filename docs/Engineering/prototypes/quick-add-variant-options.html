Quick-add × the variant flow — three ways to attach a new product to a parent

The shared quick-add sheet (_ProductSearchCreateSheet) gains a create mode shaped like the Detail page's _AddVariantForm: name + a collapsed "Override inherited defaults" disclosure. The only difference between these options is how the parent enters context so the submit can route to CreateVariantCommand.

Scenario: you're on a Take Stock walk in the Fridge. "Milk" already exists as a parent group (variants: Whole Milk, Skim Milk) and so can't itself hold stock. You're holding a carton of Oat Milk that isn't in the catalog yet.

Option A

Search-surfaced parent

Add item

  • Milk group — pick a variant
  • Whole Milk
  • Skim Milk
  • + New variant of “Milk”…
  • + Create new product “milk” (standalone)
— after choosing “New variant of Milk” —
New variant of Milk · defaults inherited from parent
Override inherited defaults

Closest mirror of the Detail page — the parent is the context you act under. Kills the collision dead-end for free (you literally can't pick the un-stockable parent; you pick a variant or add one). Search results need a grouped render + two new "create" rows.

Option B

Explicit parent picker

Add item

  • + Create new product “oat milk”
— create form —
Override inherited defaults

Form drives it — parents need not surface in search. Simplest search path; one extra control on every create. The override disclosure greys out until a parent is chosen. Collision dead-end still needs a separate guard (typing an exact parent name).

Option C

Both (hybrid)

Add item

“Milk” is a product group — pick a variant below or add a new one.
  • Milk group
  • Whole Milk
  • Skim Milk
  • + New variant of “Milk”…
  • + Create new product…
— create form (either entry point) —
Override inherited defaults

Both entry points converge on the same form + CreateVariantCommand. Most flexible (search shortcut and a fallback picker), but the largest surface to build and test, and two ways to do one thing can read as redundant.

All three submit through the same CreateVariantCommand(parentId, name, unitOverride?, categoryOverride?, locationOverride?) when a parent is set, and CreateProductCommand when standalone — so the back-end work is identical; only the picker UI differs.
